## Observation / minor finding (NOT a correctness bug in the design)
`--once` runs exactly one poll iteration: the adapter advances its **in-memory** offset but never
issues the second `getUpdates` that would ACK the batch to Telegram. So:
- a repeated `--once` re-fetches and **reprocesses** the same pending updates (duplicate replies + cost);
- in the smoke, only the latest pending message (`đã gửi`) was processed in the single batch;
  `/start` and `xin chao` were no longer pending at run time (consumed during chat_id discovery probes).
Continuous mode (`ai-dev gateway`, no `--once`) acks correctly on the next iteration, so this only
affects repeated one-shot runs. **Candidate hardening:** have `--once` flush the advanced offset
(one final `getUpdates(offset=advanced, timeout=0)`) so one-shot runs are idempotent. Deferred decision.
